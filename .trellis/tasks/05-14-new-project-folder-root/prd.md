# 新建项目弹窗：统一项目根目录（多仓同根）

## Goal

在「新建项目」时让用户**显式选择**磁盘上的文件夹作为**项目统一根目录**（`rootPath`）。项目下可以有**多个** Git 仓库（例如前端、后端子目录各一仓），且它们都位于该根之下，以便后续将 spec、rules、trellis 等收束到该根下。

用户在根下拉取新仓后，通过**「重新初始化当前项目」**一键把新仓纳入 Wise 生态（并可在保留已有 spec 的前提下重生成图谱等派生物），而不依赖日常「加仓」入口。

**已否定的方向**：不是「单仓锁定」；不是「靠应用内入口不断往项目里挂新仓」。

## User value

- 创建时即固定「工作区根」，项目层文件有唯一安置点。
- 多仓布局由用户在磁盘上组织（同根多 `.git`），而不是依赖侧栏「关联/拖入加仓」心智。
- 用户在根下拉取新仓后，可通过**一次操作**将新仓纳入当前项目的 Wise 生态（索引、工作流图谱等），且**不丢**已有项目层 spec。

## Product decisions（已确认）

- **统一根 + 硬性拦截**：凡会导致「成员仓不在同一 `rootPath` 树下」或无法为项目层 spec 选定唯一根的状态，须拦截（与先前结论一致；实现清单见后续 `design.md`）。
- **创建后不再提供「向项目新增/关联仓库」的产品入口**（无单独加仓必要）。用户若要在项目里多一个仓，在**项目根目录下**自行 `git clone` / 放置即可；应用侧**不**再维护一套「主动把游离仓挂进项目」的主流程。
- **根下新仓如何出现在 Wise 里**（扫描时机、与 `repositories`/DB 的同步方式）不在本 PRD 细化，由 `design.md` 给出可实现方案与验收子项。
- **重新初始化当前项目**：在保留**已有**项目层 spec（及用户明确视为「不可因重跑而丢」的同类产物，清单在 `design.md` 冻结）的前提下，对当前项目执行一次「生态对齐」——例如将 `rootPath` 下**新出现且尚未纳入**的 Git 仓库登记进项目成员、**重新生成**工作流图谱等派生物；用于替代「逐个加仓入口」，与「无日常加仓入口」互补。

## Confirmed facts（来自代码库）

- 新建项目 UI：`src/components/LeftSidebar/ProjectNameModals.tsx`；提交：`LeftSidebar.tsx` → `useRepositoryList.handleCreateProject`。
- 当前存在多种「把仓库挂到项目上」的路径（如 `add_repository_to_project`、关联弹窗、拖放等），与本决策冲突，实现阶段需收敛或移除对**已建项目**的加仓入口。
- `ProjectItem` 已有 `rootPath?: string` 与 `repositoryIds: number[]`。
- `create_project` 的 `root_path` 与「用户所选文件夹即根」需在实现上对齐（含无 `.trellis` 的目录）。
- 已有 `pickFolder()`、`createRepositoryFromPath` / `createRepositoryFromPathWithType`。

## Requirements

1. **新建项目**：弹窗增加「选择根目录」文件夹选择器；创建成功时 `rootPath` **必须**为所选目录（工作区根；不要求根本身为 git 根）。
2. **创建完成之后**：不提供（或不再暴露）用于「向该项目再挂一个仓库」的独立入口；侧栏/弹窗/拖放等交互中，**不得**再引导用户把游离仓**追加**进已有项目（具体涉及哪些控件以代码审计为准，在 `design.md` 列清单）。
3. **多仓**：仍允许同一 `rootPath` 下多个 Git 仓库并存，由用户在磁盘上创建/拉取；与「无加仓入口」不矛盾。
4. **违背状态**：若持久化数据中出现「某已记录成员仓路径不在项目 `rootPath` 树下」等不一致，须**硬性拦截**依赖统一根的能力，直至数据恢复一致（修复路径在实现中定义，**不**要求自动搬迁目录）。
5. **重新初始化当前项目**（命名可落地为「同步项目」「重新对齐项目」等，实现前在文案上统一）：
   - **触发**：用户显式一键触发（非静默、非定时自动全盘重跑，除非 `design.md` 另有约定）。
   - **必须保留**：当前项目下已有、位于统一根语义内的 **spec**（路径与格式以现有产品为准）；不得因重跑而覆盖或删除用户已编辑的 spec 内容。
   - **允许/应当更新**（至少包含用户已点名的方向，其余在 `design.md` 列全量清单）：例如根据**最新**成员仓集合**重新生成工作流图谱**；以及将磁盘上已存在、符合 `rootPath` 约束但尚未进入 Wise 项目生态的新仓库**纳入**（登记、索引、参与后续项目级能力）。
   - **与第 2 条关系**：不提供「零散加仓」入口，但**允许**通过本操作批量/对齐式更新「项目 ↔ 根下仓库」的绑定与派生状态。

## Acceptance Criteria

- [ ] 新建项目必须选定根目录后方可创建；`rootPath` 与所选目录一致（含无 `.trellis` 的目录）。
- [ ] 对已建立的项目：用户在日常使用中**看不到/用不了**「再往这个项目里新增一个仓库」的主路径（与旧版「关联到项目」类能力对比可回归）。
- [ ] `design.md` 中约定：用户在 `rootPath` 下新增 clone 后，Wise 如何发现并展示该仓；并按该约定编写可测验收子项。
- [ ] 处于违背状态（成员与 `rootPath` 不一致等）时：硬性拦截 + 可理解的修复引导。
- [ ] 提供「重新初始化当前项目」入口；执行后：已有项目层 **spec** 内容仍可读且未被清空覆盖（在 `design.md` 定义「spec」边界与对比方式）。
- [ ] 在同一项目中，用户于 `rootPath` 下新增 clone 后，执行该操作可将新仓纳入项目生态（成员列表或等价模型更新可测）；且工作流**图谱**按约定完成重建或等价刷新（具体步骤见 `design.md`）。

## Out of scope

- 自动搬迁磁盘上的仓库或重写用户目录结构。
- 将历史 spec/rules 从各仓自动合并到根。
- 「重新初始化」**不**等同于全盘抹掉项目：不要求删除用户数据为默认行为；除 PRD 已列的「可更新派生物」外，其余是否触碰由 `design.md` 白名单约束。
